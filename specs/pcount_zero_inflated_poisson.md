# Zero-inflated Poisson N-mixture / pcount Model

## Clean-room statement

This specification and implementation are clean-room. They use independently written model equations,
public documentation, and black-box output comparison only. Do not copy, translate, inspect, or
paraphrase R/C/C++/TMB/Stan source code from `unmarked` or any GPL package.

## Model

For site `i = 1..M` and visit `j = 1..J`:

```text
N_i ~ ZeroInflatedPoisson(lambda_i, psi)
y_ij | N_i ~ Binomial(N_i, p_ij)
log(lambda_i) = X_i beta
logit(p_ij) = W_ij detection
psi = logistic(logit_psi)
```

`lambda_i` is the Poisson mean of the non-structural-zero component. `psi` is the
structural-zero probability. The marginal mean of latent abundance is:

```text
E(N_i) = lambda_i * (1 - psi)
```

## ZIP PMF

```text
P(N_i = 0) = psi + (1 - psi) * Poisson(0 | lambda_i)
P(N_i = n) = (1 - psi) * Poisson(n | lambda_i), for n > 0
```

The implementation evaluates this in log space:

```text
n = 0: logaddexp(log(psi), log1p(-psi) + log_poisson_pmf(0, lambda))
n > 0: log1p(-psi) + log_poisson_pmf(n, lambda)
```

## Input shapes

- `y`: sites × visits count matrix, passed as `float64`; `NaN` means missing visit.
- `X`: sites × abundance-parameter design matrix. Include the intercept column explicitly.
- `W`: sites × visits × detection-parameter design tensor.
- `K`: finite upper truncation for latent abundance summation.

## Parameter ordering

```text
theta = [beta..., detection..., logit_psi]
```

## Likelihood

For one site:

```text
L_i = sum_{N=max(y_i observed)}^K ZIP(N | lambda_i, psi)
      product_{j observed} Binomial(y_ij | N, p_ij)
```

Overall log-likelihood:

```text
logL = sum_i log(L_i)
```

If a site has any positive observed count, the latent abundance sum begins above zero and the
structural-zero term contributes nothing. If a site has all zero or missing observed counts, `N=0`
is included and uses the logaddexp zero-mass expression above.

## Missing data

- `NaN` counts are skipped in the detection product.
- Non-missing counts must be non-negative integers.
- Infinite values are invalid.

## K truncation handling

`K` must be at least the maximum observed count. Larger values reduce truncation error at additional
cost. If a site has an observed count above `K`, the implementation raises a clear error.

## Identifiability caution

ZIP N-mixture models can be harder to fit and interpret because low detection probability, low
abundance, and structural zeros can all produce many observed zero counts. Users should inspect
convergence, compare against Poisson/NB alternatives, and avoid over-interpreting zero inflation from
weak data.

## Validation expectations

- Shape consistency is checked at the Python and Rust boundaries.
- `lambda_i` must be positive and finite.
- `logit_psi` must be finite and is transformed internally with a stable logistic function.
- `psi` must be in `(0, 1)` after transformation.
- Stable inverse-logit is used for detection probabilities.
- Probabilities are clamped away from exact 0 and 1 only for finite binomial logarithms.
- Cached problem objects validate counts and fixed shape data once.

## Benchmark protocol

The ZIP benchmark uses synthetic data generated by structural-zero indicators plus Poisson latent
abundance for non-structural-zero sites. Optional black-box comparison to R `unmarked::pcount(...,
mixture="ZIP")` is used for validation. Coefficient parity is reported only when the parameter names
and ordering are clear; log-likelihood parity is the primary black-box target.
